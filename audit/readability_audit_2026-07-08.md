# Readability Audit — 國中數感實驗室 (Junior Math Lab)
受眾:台灣國一學生(12–13 歲)。審查日:2026-07-08。審查者:對抗式(未參與建造)。
範圍:js/main.js(12 關 intro/formal/goals/quiz/UI)、tools/captions.json(46 caption)、index.html。
方法:讀原檔 + 用 tools/gen_narration.py 的 say() 實跑模擬 46 條 caption 的 TTS 輸出。
結論:只讀不改。以下每條附 檔案:行號 + 原句 + 一句話改法 + 信心。

課綱範圍總評(查過,沒問題):12 關全部落在國一(七上/七下)範圍內,對照 SYLLABUS.md 一致;
未出現「函數」「斜率」「一次函數」等國二以上詞彙(J7/J8 用「直線」「slope-intercept 只在程式碼註解、不外露」)。

P1 — 會造成誤解或唸錯,必修
═══════════════════════════════════════

P1-1 【TTS 括號吞字黏音】captions.json:50 (J2_4)
  原句:「換你來,之後還有第二題 2x = 6(提示:分一半)」
  say() 輸出:「…第二題 2x 等於 6提示:分一半」— 左括號被刪成空字串,"6" 與 "提示" 黏成「6提示」無停頓。
  改法:caption 內不要用括號夾提示語,改逗號:「…2x = 6,提示:分一半」。信心:高(實跑驗證)。

P1-2 【TTS 括號吞字黏音】captions.json:170,190 (J8_1, J8_4)
  J8_1 原句:「…用空心圈(開圈)標記端點」→ say():「…用空心圈開圈標記端點」(圈開圈黏連、語意重複難聽)。
  J8_4 原句:「…用閉圈,大於等於所以箭頭向右」本身 OK,但同型括號風險散佈全站。
  改法:caption 避免「詞(同義詞)」結構,擇一;或把括號改為破折號/逗號。信心:高。

P1-3 【TTS 括號吞字黏音】captions.json:210 (J9_3)
  原句:「去掉離群值(拖回 0),平均就正常了」→ say():「去掉離群值拖回 0,平均就正常了」
  「離群值」與「拖回」黏連,聽起來像「離群值拖回」是一個動作串。改法:改逗號「去掉離群值,拖回 0,…」。信心:高。

P1-4 【TTS 座標括號消失,語意變糊】captions.json:55 (J3_0)
  原句:「指令 (3, −2):第一個數管左右…」→ say():「指令 3, 負2:第一個數…」
  座標的「一組」感消失,唸成「指令三,負二」像兩個獨立指令,恰與本關要教的「有序數對是一體」相反。
  改法:caption 用「指令 x=3、y=−2」或「座標三負二」等不依賴括號的講法。信心:中高(此關教的正是有序數對)。

P1-5 【TTS 冪次唸法丟失】captions.json:115 (J5_4)
  原句:「60 = 2×2×3×5」→ say():「60 等於 2乘以2乘以3乘以5」— 尚可。但畫布上(main.js:656)結果顯示為
  「60 = 2²×3×5」用上標²,若日後有人把 caption 改成含 ² 版本,say() 會轉「平方」,唸成「2平方乘以3」語序仍可,
  但 J5 goal 文字與畫布用 2²×3×5、caption 用 2×2×3×5 兩種寫法並存 = 學生看到的和聽到的不一致。
  改法:統一「拆解過程用 2×2×3×5、最終式用 2²×3×5」並在 caption 明說「兩個 2、一個 3、一個 5」。信心:中。

P1-6 【一致性:index.html 關數與實際不符】index.html:6,7,9
  原句:標題/description/og 全寫「7 關」(如 "7 關互動數學…" line 7),但站內實為 12 關(七上5+七下7)。
  對點進來的家長/學生是錯誤預期。改法:改成「12 關」或「兩冊 12 關」。信心:高(數過 SUBJECTS b1=5, b2=7)。

P2-1 【句長過長 41 字】main.js:405 (J2.intro)
  「只要兩邊做一樣的事(各拿走一個、各加一個、各分一半),天平永遠平衡——這叫等量公理」
  一句塞:條件+三個並列例+結論+術語命名。改法:破折號處斷句,「…天平永遠平衡。這就叫等量公理。」信心:中。

P2-2 【句長過長 47 字】main.js:536 (J4.intro)
  「調寬和高,注意那條對角線:只要長方形的角落落在同一條對角線上,比就沒變——這就是等比的幾何長相」
  改法:冒號後獨立成句。「…注意那條對角線。只要角落落在同一條線上,比就沒變。」信心:中。

P2-3 【句長過長 48 字 + 術語密集】main.js:593 (J5.intro)
  「每個數都能拆成質數的乘積,而且拆法只有一種(順序不算)——這是算術基本定理,國一因倍數單元的地基」
  「算術基本定理」對國一偏抽象;此句同時給定理名+課綱定位,資訊過載。改法:先講白話結論,術語名獨立一句補充。信心:中。

P2-4 【句長過長 44 字】main.js:666 (J6.intro)
  「調兩個滑桿,看分子相乘、分母相乘為什麼是對的——因為格子總數是分母相乘,綠格數是分子相乘」
  改法:破折號斷句成兩句。信心:中。

P2-5 【術語未先解釋:離群值】main.js:958 (J9.intro) 與 formal:959
  intro 直接用「離群值」,首次出現未給白話定義(formal 才有,但 formal 預設摺疊)。國一統計單元「離群值」屬課綱邊緣,
  多數版本點到為止。改法:intro 首次出現處加一句白話「(跑得比其他資料遠很多的那一筆)」。信心:中。

P2-6 【回饋語氣:偏中性,可更鼓勵】main.js:624 (J5 controls msg)
  「${s.remain} 不能被 ${p} 整除,換一顆質數試試」— 已算溫和(有「試試」),OK。
  但 main.js:438「有一邊已經沒有砝碼可拿了」為純陳述,無引導。改法:加「換個做法看看」。信心:低。

P2-7 【回饋語氣:總測驗 quiz 用 ✗】main.js:764
  答錯顯示「✗ + why」。✗ 對 12 歲略嚴厲,但緊跟解釋且可整卷重來,尚可接受。
  makeQuiz 通過畫面「差一點,再來一輪!」(main.js:739)語氣良好。改法:可把 ✗ 換成「再看一次:」。信心:低。

P2-8 【一致性:用詞「方程式/等式」混用】main.js:405,406
  J2 intro 用「等式 = 一座平衡的天平」,formal 用「等量公理…解方程」,關卡標題用「解方程式」。
  「等式」與「方程式」在國一是不同概念(等式含恆等式),此處指的是含未知數的方程式,用「等式」易混。
  改法:intro 統一為「方程式就像一座平衡的天平」。信心:中。

P2-9 【一致性:「概率/機率」混用】main.js:320-328 (EP 表)
  EP 影片標題用「概率EP04」「正態分佈」(大陸用語),站內關卡與 caption 用「機率」「常態」(台灣用語)。
  EP 表是外連影片標題,面向國一台灣生看到「概率/正態」會突兀。改法:顯示名台灣化,或加註「(即機率)」。信心:中低
  (註:這些是連結文字,非教學正文,嚴重度較低)。

P2-10 【TTS:比號 : 依賴 G2P】captions.json:74,80 (J4_0/J4_1)
  「4:3」「8:6」的冒號不在 say() 轉換表,交給 misaki G2P 處理,唸法(四比三 vs 四冒號三)未經驗證。
  建議:實聽 audio/J4_0.mp3 確認唸成「比」;若唸歪,在 say() 加數字間 : → 「比」規則。信心:中(未實聽 mp3)。

弱模型維護陷阱(給後續 session 的警告)
═══════════════════════════════════════

W-1 【括號 = 靜音地雷】gen_narration.py:29 把 ( ) （ ）全部替換成空字串(非停頓)。
  任何維護者在 caption 裡加「說明(補充)」都會產生黏音 bug(見 P1-1~P1-4)。
  這是全站最容易被改壞的格式。建議在 captions.json 頂部或 gen_narration.py 註解明寫「caption 禁用括號,改逗號」。

W-2 【caption 與畫布/goal 文字是兩份,易漂移】captions.json 是 main.js demo() cap 的鏡像副本(手動同步)。
  改了 main.js 的 cap 但忘記重跑產字幕/重生 mp3,會導致「螢幕字幕 = 新、語音 = 舊」。
  已發現 J5 冪次寫法(2×2×3×5 vs 2²×3×5)在 caption/畫布不一致(P1-5),即此類漂移的徵兆。

W-3 【index.html 關數硬編碼】index.html:6,7,9 三處寫死「7 關」,與 SUBJECTS 陣列脫鉤(P1-6)。
  日後加關卡,這三處不會自動更新。建議註解提醒或改為動態。

W-4 【HTML 內嵌於 JS 字串】intro/formal 是含 <b><p> 的模板字串(main.js:348 等)。
  維護者若在中文標點旁誤刪 </b> 或漏跳脫,不會報錯只會顯示錯亂。全站大量此模式,改動需小心。

W-5 【say() 的 = → " 等於 " 對含 = 的口語 caption 安全,但對 <= >= 需靠 ≤≥ 符號】
  維護者若在 caption 手打「<=」而非「≤」,say() 會拆成「小於 等於」中間可能斷開。captions.json 目前正確用 ≤/≥/< 符號,勿改為 ASCII。
